# detekt 2.0.0-alpha.3 — performance regression in the ktlint ruleset path

## TL;DR

Running detekt against the same Kotlin source tree (~720 files) with the formatting / ktlint rule set produces a clear, reproducible regression on **2.0.0-alpha.3** vs **1.23.8**:

| | 1.23.8 + `detekt-formatting` | 2.0.0-alpha.3 + `ktlint-wrapper` | Ratio |
| --- | --- | --- | --- |
| Wall (median of 3) | 5.39 s | 12.28 s | **2.28×** |
| User CPU (median of 3) | 63.80 s | 138.69 s | **2.17×** |
| Peak RSS (median of 3) | 6.78 GB | 19.44 GB | **2.87×** |

Allocation flame graphs of the alpha show three frames in `org.jetbrains.kotlin.analysis.api.standalone.base.projectStructure.*` clocking ~76,000 allocations each, in lockstep with `ApplicationServiceRegistration.registerWithCustomRegistration`. None of these frames appear in the 1.23.8 profile. This points strongly at the new Kotlin Analysis API standalone backend being initialized repeatedly during a single CLI invocation rather than once per run.

The same codebase, run with the **default rule set only** (no plugins) on both versions, does *not* regress on CPU or memory — only ~22% on wall time. So the regression is specific to the ktlint plugin loading path, not the engine in general.

## 1. Reproduction

This branch ships a self-contained reproducer at `detekt-2.0-perf-repro/profile.sh`. From the repo root:

```bash
# Default: 3 cold-JVM runs of each version, formatting/ktlint workload only
./detekt-2.0-perf-repro/profile.sh

# Full reproducer (capture flame graphs + run negative controls):
./detekt-2.0-perf-repro/profile.sh --flames --negative-control --all-rules
```

The script:

1. Downloads (once, ~155 MB) the four CLI/plugin jars to `detekt-2.0-perf-repro/jars/`:
   - `detekt-cli-1.23.8-all.jar`, `detekt-formatting-1.23.8.jar`
   - `detekt-cli-2.0.0-alpha.3-all.jar`, `detekt-rules-ktlint-wrapper-2.0.0-alpha.3.jar`, `ktlint-repackage-2.0.0-alpha.3-all.jar`
2. Enumerates every `<module>/src/{main,test}/kotlin` plus `documentation-website/Writerside/snippets` and `samples` (38 roots, ~720 `.kt` files) into `inputs-{comma,colon}.txt`. The CLI's input separator changed between versions (`,` → `:`).
3. Runs each version's CLI directly under `/usr/bin/time -l` (macOS) or `-v` (Linux), captures wall time / user CPU / peak RSS, and prints per-run summaries.
4. With `--flames`: attaches async-profiler via `-agentpath:libasyncProfiler.{dylib,so}=start,event=cpu,file=…` and again with `event=alloc`, producing four HTML flame graphs.
5. With `--negative-control` / `--all-rules`: runs additional configurations to confirm the regression is specific to the ktlint plugin path (results in §2.2 / §2.3 of this report).

A single-run smoke test (`--runs 1`) reproduced the regression cleanly on the report author's machine: 5.10 s / 59.6 s CPU / 6.07 GB on 1.23.8 vs 10.86 s / 125.4 s CPU / 18.96 GB on 2.0.0-alpha.3.

### Environment used for the measurements in this report

| | |
| --- | --- |
| Host | macOS, Darwin 25.4.0 |
| JDK | OpenJDK / JBR 17.0.14, 64-bit |

### Equivalent raw command lines

In case you want to drive the runs by hand, this is exactly what `profile.sh` invokes:

```
# 1.23.8
java -jar detekt-cli-1.23.8-all.jar \
  --parallel \
  --input "$(cat inputs-comma.txt)" \
  --plugins detekt-formatting-1.23.8.jar

# 2.0.0-alpha.3 (ktlint-repackage manually fetched —
# the cli-all fat jar does not bundle it)
java -jar detekt-cli-2.0.0-alpha.3-all.jar \
  --parallel \
  --input "$(cat inputs-colon.txt)" \
  --plugins detekt-rules-ktlint-wrapper-2.0.0-alpha.3.jar:ktlint-repackage-2.0.0-alpha.3-all.jar
```

`inputs-comma.txt` and `inputs-colon.txt` enumerate the same 38 source roots in this repo, separated by `,` (1.x) and `:` (2.x). They're regenerated by `profile.sh` on every run.

## 2. Measurements

### 2.1 Formatting / ktlint workload (the regressing one)

3 cold-JVM runs each. `/usr/bin/time -l` for peak RSS.

| run | 1.23.8 + formatting | 2.0.0-alpha.3 + ktlint-wrapper |
| --- | --- | --- |
| 1 | 5.41 s / 63.80 s CPU / 6.96 GB | 12.28 s / 138.69 s CPU / 18.55 GB |
| 2 | 5.15 s / 63.16 s CPU / 6.78 GB | 11.70 s / 138.56 s CPU / 21.49 GB |
| 3 | 5.39 s / 65.30 s CPU / 6.56 GB | 12.85 s / 146.90 s CPU / 19.44 GB |
| **median** | **5.39 s / 63.80 s CPU / 6.78 GB** | **12.28 s / 138.69 s CPU / 19.44 GB** |
| **ratio** | — | **2.28× / 2.17× / 2.87×** |

Under the same default `-Xmx`, alpha triggers more Young GCs and routinely peaks at 18–22 GB resident before collection. 1.23.8 stays around 6–7 GB.

### 2.2 Default rules only (negative control)

Same input, no `--plugins`, no `--config`. Engine defaults only.

| | 1.23.8 | 2.0.0-alpha.3 | Ratio |
| --- | --- | --- | --- |
| Wall (median of 3) | 2.88 s | 3.50 s | 1.22× |
| User CPU (median of 3) | 31.91 s | 29.59 s | 0.93× |
| Peak RSS (median of 3) | 1.64 GB | 1.44 GB | 0.88× |

The engine itself does not regress on CPU or memory — only wall time, by ~22%. Confirms the regression lives specifically in the ktlint-wrapper plugin loading / execution path.

### 2.3 `--all-rules` (broader engine surface, still no plugins)

| | 1.23.8 | 2.0.0-alpha.3 | Ratio |
| --- | --- | --- | --- |
| Wall | 3.23 s | 4.30 s | 1.33× |
| User CPU | 32.60 s | 35.78 s | 1.10× |
| Peak RSS | 1.95 GB | 1.87 GB | 0.96× |

Even with all built-in rules enabled, the engine alone is within 10% on CPU and slightly *better* on memory. The 2-3× regression in §2.1 is therefore not explained by "more rules" or "stricter defaults" — it requires the ktlint plugin to be loaded.

### 3.5 The smoking gun

Three frames in `org.jetbrains.kotlin.analysis.api.standalone.base.projectStructure.*` clock **76,333 / 76,344 / 76,443** allocations respectively — three counters in lockstep, ~106 per Kotlin source file (76,000 / 720). 1.23.x doesn't touch this package at all (it uses plain `KotlinCoreApplicationEnvironment`, `create()`'d once).

`ApplicationServiceRegistration.registerWithCustomRegistration` walking PSI elements per registration explains:

- the 3.6% allocation share spent in `KtParameterList.<init>` and 2.7% in `KtVisitor.visitImportDirective` — PSI element construction during repeated registration;
- the 1.0% in `LinkedList` allocations — service indices being rebuilt;
- the `ClassLoader.getClassLoadingLock` and `__semwait_signal` time in the CPU profile — thread-safe singleton initialization being contested across the worker pool.

The same `StandaloneProjectFactory.*` and `ApplicationServiceRegistration.*` frames are **absent from the default-rules-only alpha alloc profile** (`profiles-2.0.0-alpha.3/alloc.html`). They show up only when the ktlint-wrapper plugin is loaded. This narrows the culprit to the wrapper plugin's loading or per-rule initialization path triggering the standalone Analysis API project setup.

Suspected cause: detekt 2.0 is initializing a `StandaloneAnalysisAPISession` (or equivalent) per ruleset / per file / per rule batch, instead of once per CLI invocation.
