# Walkthrough Guide — Exposed Gradle Plugin

Work through the scenarios in order. Each scenario is intended to end with **two commits**:

1. The **schema change** (edit to `src/main/kotlin/com/example/tables/*` and/or `build.gradle.kts`).
2. The **generated migration** (`src/main/resources/db/migration/*.sql`) — the artifact of running `./gradlew generateMigrations`.

Between scenarios, **apply the generated SQL to `data/mydb`** via IntelliJ's Database tool so the next diff is against the new baseline. See `README.md` for setup.

---

## Pre-flight

```bash
cd samples/exposed-gradle-plugin-test
./gradlew generateMigrations --rerun
# Expected:
#   # Exposed Migrations Generated 2 migrations:
#     * V<ts>__CREATE_TABLE_USERS.sql
#     * V<ts>__CREATE_TABLE_CITIES.sql
```

If the plugin resolves and produces those two files against the (implicitly created) empty H2 DB, you're good.

## Scenario 1 — Baseline: two independent `CREATE TABLE`s

**Exercises:** happy path, `CREATE TABLE` branch of `DescriptionGenerator`, absence of FKs.

**Setup:** none — the sample already contains `Users` and `Cities` with no FK.

**Commands:**
```bash
./gradlew generateMigrations
```

**Expected:**
- Two files under `src/main/resources/db/migration/`:
  - `V<ts>__CREATE_TABLE_USERS.sql`
  - `V<ts>__CREATE_TABLE_CITIES.sql`
- Each contains a single `CREATE TABLE IF NOT EXISTS <name> (...)` statement.
- The `<ts>` is a 14-digit `YYYYMMDDHHMMSS` (same for both since they generate in the same second).

**Apply:** execute both files against `data/mydb` in IntelliJ.

**Commit:** `git commit -am "scenario 1: baseline — two CREATE TABLEs"`

## Scenario 2 — Add a foreign key

**Exercises:** `ALTER TABLE ... ADD CONSTRAINT` branch of `DescriptionGenerator`; `SchemaUtils.sortTablesByReferences`.

**Edit `Users.kt`:**
```kotlin
object Users : Table("users") {
    val id = integer("id").autoIncrement()
    val name = varchar("name", 255)
    val email = varchar("email", 255)
    val cityId = integer("city_id").references(Cities.id).nullable()

    override val primaryKey = PrimaryKey(id)
}
```

**Commands:**
```bash
./gradlew generateMigrations
```

**Expected:**
- 1 new file `V<ts>__ALTER_TABLE_USERS_ADD_CONSTRAINT_<fk_name>.sql` (fk_name auto-generated by Exposed, something like `FK_USERS_CITY_ID__ID`).
- File contains `ALTER TABLE USERS ADD COLUMN CITY_ID INT NULL;` followed by `ALTER TABLE USERS ADD CONSTRAINT ... FOREIGN KEY (CITY_ID) REFERENCES CITIES(ID) ...;`.

**Note:** the plugin uses the **first statement** to build the filename. If the first statement is the ADD COLUMN rather than ADD CONSTRAINT, the filename will be `V<ts>__ALTER_TABLE_USERS.sql` instead. Record which one you see — both behaviors are informative.

**Apply, commit:** as before.

## Scenario 5 — Add a unique index

**Exercises:** `CREATE INDEX` branch of `DescriptionGenerator`, `extractIndexInfo`.

**Edit `Users.kt`:** add `init { uniqueIndex(email) }` inside the `Users` object.

```kotlin
object Users : Table("users") {
    val id = integer("id").autoIncrement()
    val name = varchar("name", 255)
    val email = varchar("email", 255)
    val cityId = integer("city_id").references(Cities.id).nullable()

    override val primaryKey = PrimaryKey(id)

    init {
        uniqueIndex(email)
    }
}
```

```bash
./gradlew generateMigrations
```

**Expected:**
- `V<ts>__CREATE_INDEX_<index_name>_ON_USERS.sql` containing `CREATE UNIQUE INDEX USERS_EMAIL_UNIQUE ON USERS (EMAIL);` (or similar name auto-generated by Exposed).

**Apply, commit.**

## Scenario 6 — Drop a table (and its FK first)

**Exercises:** `DROP TABLE` branch, and exercises the ordering of drop statements.

**Edit `Users.kt`:** remove the `cityId` column (the FK would prevent dropping `Cities` otherwise).

**Edit `Cities.kt`:** delete the file entirely, or `rm src/main/kotlin/com/example/tables/Cities.kt`.

```bash
./gradlew generateMigrations
```

**Expected:**
- A migration that drops the `cityId` column / FK on Users.
- A migration `V<ts>__DROP_TABLE_CITIES.sql` containing `DROP TABLE CITIES;`.
- May end up as 1 file or 2 files depending on how MigrationUtils groups the statements — record either way.

**Apply, commit.**

## Scenario 7 — Switch to `MAJOR_MINOR`

**Exercises:** `VersionFormat.MAJOR_MINOR` + `findNextMajor`. Confirms that switching version formats correctly ignores existing files written in a different format.

**Edit `build.gradle.kts`:** uncomment the `fileVersionFormat` line and change to `MAJOR_MINOR`:
```kotlin
fileVersionFormat.set(VersionFormat.MAJOR_MINOR)
```

Make any trivial schema change (e.g., add `val description = text("description").nullable()` to `Users`).

```bash
./gradlew generateMigrations
```

**Expected:**
- A file named `V1_1__ALTER_TABLE_USERS.sql`.
- The major/minor versions start from `1`. The earlier `TIMESTAMP_ONLY` files (`V<14-digit-ts>__...`) are **not** misread as major versions — `findNextMajor` only matches the format-specific patterns, so a 14-digit timestamp is never treated as a major version number.

**Apply, commit.**

## Scenario 8 — `MAJOR_ONLY` in a pristine directory

**Exercises:** `MAJOR_ONLY` start-from-1 path when no prior matching files exist.

**Setup:**
```bash
rm src/main/resources/db/migration/*.sql
touch src/main/resources/db/migration/.gitkeep
```

Switch `fileVersionFormat` to `VersionFormat.MAJOR_ONLY`.

Revert scenario 7's `description` column (or keep it — doesn't matter; we just need a non-empty diff).

```bash
./gradlew generateMigrations
```

**Expected:**
- `V1__<desc>.sql` (starting from `1`, not padded).
- If there are multiple statements, files are `V1__`, `V2__`, `V3__`... per statement.

**Apply, commit.**

## Scenario 9 — Custom prefix, separator, case, extension

**Exercises:** `filePrefix`, `fileSeparator`, `useUpperCaseDescription`, `fileExtension` properties simultaneously.

**Edit `build.gradle.kts`:**
```kotlin
filePrefix.set("M")
fileVersionFormat.set(VersionFormat.TIMESTAMP_ONLY)  // reset
fileSeparator.set("-")
useUpperCaseDescription.set(false)
fileExtension.set(".ddl")
```

Make any schema change (e.g., add `val bio = text("bio").nullable()` to Users).

```bash
./gradlew generateMigrations
```

**Expected:**
- A file matching `M<ts>-alter_table_users.ddl` — lowercase description, custom separator, custom extension, custom prefix.

**Apply, commit.** (Apply against H2 even though the extension is `.ddl` — it's still SQL.)

## Scenario 10 — `--filename` CLI override

**Exercises:** `@Option` command-line arg and single-file bundling path.

**Revert `build.gradle.kts`** to defaults (re-comment the customized properties).

Make **two pending schema changes** (so the plugin would otherwise generate 2 files):
- Add `val nickname = varchar("nickname", 64).nullable()` to `Users`.
- Add a new table `Orders` in `src/main/kotlin/com/example/tables/Orders.kt`:
  ```kotlin
  package com.example.tables

  import org.jetbrains.exposed.v1.core.Table

  object Orders : Table("orders") {
      val id = integer("id").autoIncrement()
      val userId = integer("user_id").references(Users.id)

      override val primaryKey = PrimaryKey(id)
  }
  ```

```bash
./gradlew generateMigrations --filename=V999__snapshot.sql
```

**Expected:**
- **One** file `V999__snapshot.sql`.
- Contains **all** statements (ALTER USERS + CREATE TABLE ORDERS + FK) concatenated with `;\n`.
- No other files added.

**Apply, commit.**

## Scenario 11 — Error path: missing mandatory config

**Exercises:** Gradle's `@Input` validation for `tablesPackage` (no `@Optional`).

**Edit `build.gradle.kts`:** comment out the `tablesPackage.set(...)` line (keep everything else intact).

```bash
./gradlew generateMigrations
```

**Expected:**
- Task fails with `> A problem was found with the configuration of task ':generateMigrations' of type 'GenerateMigrationsTask'. ... this property isn't marked as optional and no value has been configured`.
- Exit code non-zero.

**Revert, no commit needed** (or commit the revert — your call).
